## Идея 1 — Token Economics Calculator

**Статус:** 💡 Raw idea (2026-05-30)
**Приоритет:** средний — полезен после накопления данных в `llm_usage`

### Что это

Интерактивный веб-калькулятор стоимости использования Claude API. Два режима:
- **Аналитический** — показывает реальные данные из `llm_usage` (наш pipeline)
- **Предиктивный** — позволяет вводить параметры и считать прогноз

Старт: внутренняя страница `/economics` в трекере agent-hub на реальных данных.
Потенциал: отдельный публичный инструмент для разработчиков, строящих на Claude API.

---

### Ценность

**Для нас (agent-hub):**
- Понимать где и сколько тратим по фазам
- Видеть эффективность кэша в реальном времени
- Оптимизировать budget_tokens на thinking (сейчас 10k, реально используем ~1k)
- Прогнозировать месячный бюджет при росте объёма вакансий

**Для разработчиков, строящих на Claude API:**
- Оценить COGS до написания кода
- Выбрать модель (Opus / Sonnet / Haiku) с учётом реального cost/quality trade-off
- Обосновать бюджет перед командой или инвестором
- Найти "дорогие" места в pipeline и оптимизировать

---

### Функции — аналитический модуль (на реальных данных)

#### Dashboard метрик (из `llm_usage` DB)
- **Cost per phase** — средняя, мин, макс стоимость каждой фазы (phase1…phase4)
- **Cost per vacancy** — полная стоимость pipeline на одну вакансию
- **Cache efficiency** — `cache_read_tokens / (input_tokens + cache_read_tokens)` в %
- **Thinking utilization** — `thinking_tokens / budget_tokens` в % по фазам
- **Latency по фазам** — `elapsed_ms` средний, перцентили p50/p90
- **Model distribution** — если используется несколько моделей, разбивка по стоимости
- **Daily/weekly spend** — динамика расходов во времени (график)
- **Input breakdown** — profile_tokens vs prompt_tokens vs user_tokens (из наших оценок)

#### Детализация по вакансии
- Выбрать вакансию → показать все 5 фаз с полным breakdown токенов и стоимости
- Сравнение прогонов одной вакансии (v1 vs v2 vs v4)

### Функции — предиктивный калькулятор

#### Быстрый расчёт (Quick Estimate)
Форма:
- System prompt размер (chars или tokens)
- User message размер
- Ожидаемый размер ответа
- Модель (Opus / Sonnet / Haiku)
- Кэш включён? (да/нет + размер кэшируемого блока)
- Extended Thinking? (да/нет + budget_tokens)

Вывод:
- Стоимость одного вызова ($)
- Стоимость 100 / 1 000 / 10 000 вызовов
- С кэшем vs без кэша — разница

#### Pipeline-режим (Multi-step)
- Описать pipeline из N шагов
- Каждый шаг: название, модель, размеры I/O, кэш, thinking
- Итог: стоимость одного "юнита" (документ, вакансия, сессия)
- Pre-filled шаблон: наш CV pipeline (5 фаз с реальными средними значениями)

#### Thinking-оптимизатор
- Ввести текущий budget_tokens и среднюю утилизацию (из наших данных)
- Показать: "вы платите за 10 000, используете 1 043 — можно снизить до 2 000 и сэкономить X%"
- Рекомендуемый budget_tokens на основе перцентиля p95 фактического использования

#### Volume Planner
- Режим A: месячный бюджет $ → сколько юнитов можно обработать
- Режим B: планируемый объём → прогноз месячного бюджета
- Режим C: целевая маржа + планируемая цена → максимально допустимый COGS

#### Model Comparator
- Одинаковые параметры → показать Opus vs Sonnet vs Haiku
- Разница в цене + качественная оценка (based on known benchmarks)
- Recommendation: "для вашего use case оптимален Sonnet"

#### Cache Analyzer
- Ввести: размер кэшируемого блока, количество вызовов в сессии, TTL
- Показать: стоимость с кэшем vs без, savings в %, breakeven point (при скольких вызовах кэш окупается)
- Наш кейс: PROFILE.md 1780 tok → cache_write $0.006 + N × cache_read $0.0005 vs N × input $0.005

### Функции — unit economics для SaaS

Для разработчиков, монетизирующих продукт на Claude API:

#### COGS Calculator
- Ввести pipeline → получить COGS на юнит
- Добавить infra costs (hosting, DB, etc.)
- Total COGS per unit

#### Pricing Simulator
- Задать target gross margin (%)
- Получить минимальную цену за юнит
- Таблица: разные ценовые модели (pay-per-use, подписка с лимитом)
- Breakeven по подписке: при каком потреблении подписка убыточна

#### Margin Analyzer (на реальных данных)
- Если задать текущую цену продукта → показать фактическую маржу
- Динамика: как меняется маржа при росте объёма (эффект масштаба кэша)

### Технический стек (для внутренней версии)

- **Backend:** FastAPI endpoint `/economics` + `/economics/api/data` (JSON)
- **Data source:** `llm_usage` таблица в SQLite (уже наполняется)
- **Frontend:** HTMX + Chart.js (как трекер) — без лишнего JS фреймворка
- **Калькулятор:** vanilla JS для интерактивности форм, пересчёт на клиенте
- **Графики:** Chart.js (line для динамики, bar для фаз, doughnut для breakdown)

### Открытые вопросы

1. **Accuracy thinking_tokens:** сейчас оцениваем через `len(thinking_text) // 4` — достаточно ли точно для аналитики?
2. **Multi-project:** если вырастет в публичный инструмент — нужна аутентификация и изоляция данных
3. **Benchmarks качества:** как количественно выразить "Sonnet хуже Opus на X%"? Нет объективной метрики
4. **Cache TTL:** ephemeral_5m — данные устаревают через 5 минут. При долгих pipeline нужно учитывать потери кэша
5. **Batch API:** 50% discount для async — стоит показывать как опцию в калькуляторе
6. **Выделение в отдельный продукт:** нужен ли онбординг, API key от пользователя, или работает на наших данных как демо?
