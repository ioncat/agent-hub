# Repositioning Pivot Plan

**Date:** 2026-05-31  
**Status:** Active  
**Scope:** Full restructure from `agent-hub` generic orchestrator → Career Agent focused vertical service

---

## Context

The original design treated this as a generic multi-agent orchestration platform. After building and running the CV pipeline end-to-end, the product identity became clear: this is a **focused vertical service for PdM/PO/PM job search** — a job counselor, not a platform showcase.

Two consequences follow:
1. **Product:** tight, purpose-built pipeline is a feature, not a limitation. ICP = PdM/PO/PM in active job search.
2. **Architecture:** three previously external repos (`knowledge-mirror-parser`, `callback-cv`, `job-board-monitor`) must move inside as `services/`. Nothing user-built lives outside. Audit before migrating — strip dead code, make each component organic.

Full ADR: `ARCHITECTURE.md → Design Decisions Log → 2026-05-31`.

---

## What Changes

| Area | Before | After |
|------|--------|-------|
| Product identity | Generic AI agent orchestrator | Focused job counselor for PdM/PO/PM |
| External repos | 3 repos depended on externally | All move inside as `services/` |
| Profile storage | Hand-edited `PROFILE.md` file | Generated by onboarding, stored in SQLite |
| PDF generation | `callback-cv` subprocess + foreign venv | `services/pdf/` FastAPI HTTP service |
| Job discovery | File polling `seen_jobs.json` | Webhook push `POST /api/new-vacancy` |
| User model | Single hardcoded `TELEGRAM_CHAT_ID` | Multi-user: `users` table, `user_id` everywhere |
| UI channels | Telegram only | Channel-agnostic adapter pattern (Telegram primary, PWA next) |
| AI interoperability | None | MCP server + REST API (planned) |

## Service Audit

### callback-cv

**What agent-hub uses:**
- `cv_to_pdf.py` — Markdown → PDF renderer (fpdf2 + Calibri fonts)
- `skill/PROFILE.md` — candidate profile (to be replaced by onboarding)

**What to keep:**
- Core rendering: `md_to_pdf()`, `CVDocument` class, `render_md()` — ~260 lines
- `fonts/` — Calibri regular, bold, italic

**What to cut:**
- `--here` batch flag and batch loop
- `tracker.py` subprocess call at end of `__main__` (callback-cv internal, not ours)
- All other files in the repo

**Migration:** Extract rendering core → `services/pdf/render.py`. Wrap in FastAPI `POST /render` (accepts markdown text, returns PDF bytes). `CVAdapter` switches subprocess → HTTP. Tools layer unchanged.

**Complexity:** 🟢 Low

---

### knowledge-mirror-parser

**What agent-hub uses:**
- `POST /parse` endpoint — fetch URL → clean Markdown
- `fetch()` HTTP function with retry logic
- Site configs for `djinni.co` + `jobs.dou.ua`

**What to keep:**
- `api.py` — FastAPI endpoint (130 lines, already clean)
- `crawler.py` — `fetch()`, `_build_session()`, `_random_headers()`, `_polite_delay()` only
- `config.py` — `djinni.co` + `jobs.dou.ua` configs only

**What to cut:**
- `database.py` — sitemap cache SQLite (not used by agent-hub)
- `processor.py` — batch article processing (not used)
- `main.py` — CLI entry point for batch scraping (not used)
- `discover_urls()`, `_parse_sitemap_xml()` — sitemap batch discovery (not used)
- `login()` — WordPress auth (not used, login_url=None for our sites)
- `download_binary()` — asset downloader (not used)
- `gopractice.ru` site config — knowledge mirror, not job board

**Migration:** Move stripped source → `services/parser/`. `KMPAdapter` unchanged. Update docker-compose build context.

**Complexity:** 🟢 Low (source move + delete ~60% of files)

---

### job-board-monitor

**What agent-hub uses:**
- `seen_jobs.json` — polled by `core/rss_watcher.py` to detect new vacancies

**What to keep:**
- Full `monitor.py` RSS polling logic
- Retry/delivery state machine (`seen_jobs.json` schema)
- `feeds.json` config

**What changes:**
- Step 1 (move): relocate to `services/job-monitor/`, no functional change
- Step 2 (redesign): replace `send_telegram()` → `POST /api/new-vacancy` on Career Agent. `rss_watcher.py` becomes webhook receiver endpoint in `web/api.py`.

**Complexity:** 🟡 Medium (Step 1 trivial; Step 2 requires redesign of integration boundary)

## Development Phases

```
Phase 1 — Multi-user data model     ← start here (additive, safe)
Phase 2 — services/pdf/             ← independent
Phase 3 — services/parser/          ← independent
Phase 4 — services/job-monitor/     ← independent
Phase 5 — Onboarding                ← depends on Phase 1
Phase 6 — Rename                    ← anytime, non-blocking
```

**Critical path: Phase 1 → Phase 5**

Phases 2, 3, 4 are independent of each other and of the critical path.

---

### Phase 1 — Multi-user data model
**Goal:** Design for multi-user from the start. Cheap now, expensive to retrofit.

- [ ] `db/schema.sql` — add `users` table: `id`, `telegram_chat_id`, `name`, `created_at`
- [ ] Add `user_id` FK to `vacancies`, `llm_usage` (nullable for migration, then default=1)
- [ ] `db/database.py` — all queries accept and scope by `user_id`
- [ ] `core/deps.py` — `AgentDeps` carries `user_id`
- [ ] `core/settings.py` — `TELEGRAM_CHAT_ID` creates default user on first run, not a global singleton
- [ ] Migration: existing rows get `user_id=1`
- [ ] Tests: all DB tests pass with `user_id` param

**Does not break:** pipeline phases, LLM client, adapters, web tracker

### Phase 5 — Onboarding: PDF → Interview → Profile
**Goal:** Replace hand-edited PROFILE.md with a generated, structured profile per user.  
**Depends on:** Phase 1 (user_id in DB)

- [ ] Telegram: `/start` flow + PDF upload handler
- [ ] PDF → Markdown: `pypdf` or `pdfminer` extraction
- [ ] LLM: analyze candidate profile → generate personalized PM interview (Delivery/Discovery framing, archetype, gaps)
- [ ] Multi-turn conversation: Telegram inline keyboard + text input, LLM extracts experience depth
- [ ] Profile generation: interview transcript → structured profile stored in `users` table (JSON field or separate `profiles` table)
- [ ] Multi-pass: `/update_profile` command — re-interview, enrich existing profile
- [ ] `core/llm_client.py` — profile loaded from DB instead of `PROFILE.md` file
- [ ] Remove `PROFILE_MD_PATH` from settings
